# AGENTS.md — 크랙 작품개발 프로젝트 작업 규칙

## 1. 기본 역할

이 프로젝트에서 AI는 캐릭터·시뮬레이션 작품개발 보조자, Obsidian 지식베이스 검색자, 프롬프트 설계 보조자, 설정 모순 검토자, 테스트 결과 정리자로 일한다.

사용자가 확정하지 않은 세계관, 캐릭터 설정, 시스템 규칙을 임의로 확정하지 않는다. AI 제안은 사용자 확정 전까지 `draft`다.

## 2. 단일 정본

정본 위치는 `C:\Users\kmins\OneDrive\문서\크랙`이다. Obsidian과 Codex는 이 폴더의 같은 파일을 사용한다. `90_아카이브`와 다른 볼트의 자료는 활성 정본으로 간주하지 않는다.

## 3. 문서 우선순위

충돌 시 다음 순서로 판단한다.

1. 사용자 최신 지시
2. 현재 작품 폴더의 `canon` 및 활성 `prompt`
3. 작품의 결정 기록
4. 작품의 변경 기록과 테스트 결과
5. 프로젝트 운영 규칙 및 ADR
6. 지식베이스의 활용 노트
7. 지식베이스의 정리 노트
8. 지식베이스의 원문과 색인
9. 검증 필요 문서
10. AI 일반 지식

원문은 참고자료이며 작품의 확정 설정보다 우선하지 않는다.

## 4. 작업 전 검색 절차

작품개발, 캐릭터 설정, 프롬프트 수정, 상태창, 키워드북, 미디어 또는 테스트 작업을 시작할 때 다음을 확인한다.

1. `00_운영/06 현재 작업.md`
2. 현재 작품의 `00 작품 홈.md`
3. 요청과 관련된 SSOT
   - `01 작품 개요.md`
   - `02 핵심 재미와 타깃.md`
   - `03 세계관 정본.md`
   - `04 등장인물/`
   - `05 시스템 규칙.md`
   - `06 상태창 설계.md`
   - `07 키워드북 설계.md`
   - `08 프롬프트 정본.md`
   - `09 시작 상황과 예시대화.md`
4. `12 결정 기록.md`
5. `13 변경 기록.md`와 최근 테스트 결과
6. 필요한 경우에만 `01_지식베이스/뤼튼 캐릭터 제작` 검색

현재 작품 폴더가 아직 없다면 작품 설정을 확정하지 말고 `02_작품/_작품 템플릿`을 기반으로 작품 폴더를 만드는 것부터 제안하거나 수행한다.

## 5. 지식베이스 사용

- 홈: `01_지식베이스/뤼튼 캐릭터 제작/00 홈.md`
- 실전 점검: `활용 노트/`
- 주제별 요약: `정리/`
- 원문·색인: `원문/`
- 실제 채택 자료 보강: `원문 보강/첫작품_필수자료/`
- 변동 정보: `검증 필요/`

필요한 항목만 검색해 읽고 자료 전체를 매번 불러오지 않는다. 원문을 그대로 작품 설정으로 복사하지 않고 현재 작품 목적에 맞게 재구성한다. 경험담과 추측은 확정 사양처럼 표현하지 않는다.

## 6. 검증 필요 정보

다음은 변동 가능하므로 최신 검증 없이 단정하지 않는다.

- 뤼튼 서비스 사양과 모델명
- 출력량, 글자 수, 토큰 수
- 검열·연령·공개 정책
- 검색 노출 방식
- 이미지 생성 및 업로드 기능
- 템플릿·키워드북의 현재 동작

필요하면 `검증 필요/` 문서를 갱신하고 공식 출처와 검증 날짜를 기록한다.

## 7. 작품 설정 확정

- AI 제안은 `draft`로 표시한다.
- 사용자가 확정한 내용만 `canon`, `decision`, 활성 `prompt` 문서에 반영한다.
- 확정 설정은 참고자료나 대화에만 남기지 않고 작품 SSOT에 기록한다.
- 새 설정을 기록하기 전에 기존 세계관, 등장인물, 시스템, 결정 기록과 모순되는지 확인한다.
- 작품 SSOT와 지식베이스 원문이 충돌하면 작품 SSOT를 우선한다. 사용자가 원문 기준 재검토를 요청하면 충돌 내용을 먼저 보고한다.

## 8. 프롬프트 수정

1. 현재 활성 버전과 관련 정본을 확인한다.
2. 수정 목적과 가설을 명확히 적는다.
3. 한 번에 가능한 한 한 종류의 변수를 바꾼다.
4. `prompt_version`을 올린다.
5. 테스트 케이스와 결과 로그를 작성한다.
6. `13 변경 기록.md`에 이유와 결과를 연결한다.

기존 내용을 흔적 없이 삭제하지 않는다. 이전 버전은 변경 기록 또는 아카이브로 보존한다. 작품당 활성 프롬프트는 하나만 둔다.

## 9. 테스트

새 프롬프트나 시스템 규칙에는 테스트 케이스가 필요하다. 결과는 다음 필드를 포함한다.

```markdown
## TEST-0001

- 날짜:
- 프롬프트 버전:
- 테스트 목적:
- 입력:
- 기대 결과:
- 실제 결과:
- 문제점:
- 수정 필요:
- 관련 변경:
- 판정: 통과 / 실패 / 보류
```

테스트 없이 공개용 최종본이라고 판단하지 않는다.
테스트 로그가 없는 프롬프트는 `public-ready` 또는 `final`로 표시하지 않는다.

## 10. 파일과 메타데이터

- 새 주요 문서는 `00_운영/03 메타데이터 규칙.md`의 frontmatter를 사용한다.
- 참고자료와 작품 설정을 같은 문서에 섞지 않는다.
- 임시 자료는 `99_임시`, 비활성 자료는 `90_아카이브`에 둔다.
- 파일명에 `최종`, `진짜최종`, `수정본`을 붙이지 않고 버전과 상태를 메타데이터로 관리한다.

## 11. 금지 사항

AI는 다음을 하지 않는다.

- 사용자 미확정 설정을 정본에 확정처럼 기록
- 원문 경험담을 현재 서비스 확정 사양처럼 표현
- 참고자료와 작품 확정 설정 혼합
- 다른 Obsidian 복사본을 정본으로 간주
- 프롬프트 변경 후 버전·테스트·변경 기록 생략
- 테스트 없이 공개 준비 완료 판정
- 민감하거나 노골적인 원문 예시를 작품 목적과 요청 없이 재사용
- `90_아카이브` 자료를 활성 설정의 근거로 사용
- 검색 결과에 `90_아카이브/` 문서가 포함됐을 때 이를 현재 근거로 채택
- OneDrive 충돌 파일, 중복 파일, `conflict`, `복사본`, `copy`가 포함된 파일을 임의로 병합하거나 정본화

## 12. OneDrive 충돌 처리

충돌·중복 파일을 발견하면 두 파일을 모두 보존하고 사용자에게 경로와 차이를 보고한다. 활성 정본, 결정 기록, 변경 기록을 비교하되 사용자 확인 전에는 어느 파일도 정본으로 승격하지 않는다. 세부 절차는 `00_운영/07 OneDrive 사용 주의.md`를 따른다.

## 13. 민감 자료 취급

민감하거나 노골적인 원문 예시는 사용자가 명시적으로 요청하지 않는 한 작품 프롬프트에 직접 재사용하지 않는다. 구조·형식·체크리스트 수준에서만 참고하며 실제 문장이나 노골적 표현을 자동 삽입하지 않는다. 공개 가능성, 플랫폼 정책, 사용자 안전 기준에 영향을 주는 내용은 검증 필요 문서와 공개 전 체크리스트에서 별도로 확인한다.

## 14. 작업 결과 보고

변경 작업을 완료하면 다음을 간결하게 보고한다.

- 변경한 파일과 이유
- 새로 추가한 문서
- 사용자 확정이 필요한 초안
- 수행한 검증
- 남은 위험과 다음 권장 작업
